# MaterialPlanningSystem.md — Interior Material Planning System

## What This Project Is

An interior material planning tool that helps users build a harmonious kitchen/dining moodboard. Users are not designers — they should never need to think in terms of "warm undertone" or "cool neutral." They just make intuitive choices and the system guarantees the result is harmonious.

The core product challenge is: **hundreds of real materials need to feel like a small, curated, personal selection to the user.**

---

## The Four-Layer Architecture

This is the most important thing to understand about the system. Every layer has a specific job. Do not collapse layers or let them bleed into each other.

```
Layer 1: Vibe Tag
Layer 2: Archetype
Layer 3: Collection
Layer 4: Real Materials
```

### Layer 1: Vibe Tag
**Job:** Reduce overwhelm at the very start. Capture mood, not material preference.

The user picks one of three vibes before seeing any materials:

```typescript
type VibeTag = "light-and-airy" | "warm-and-grounded" | "bold-and-moody";
```

- `light-and-airy` → value: light, tone: cool/neutral
- `warm-and-grounded` → value: light-medium, tone: warm/natural
- `bold-and-moody` → value: medium-dark, tone: any, contrast-driven

The vibe tag pre-filters which collections are candidates. It does not lock the user in — it just narrows the starting point.

---

### Layer 2: Archetype
**Job:** Let the user make intuitive visual choices. Simple, beautiful, and limited in number.

This is what the user sees and picks in the moodboard builder. An archetype represents a *type* of material — not a specific product, brand, or exact tone. Archetype images should be beautiful and evocative. For most archetypes the image is tonally ambiguous — the exact warm/cool variant is resolved by collection matching. Bold colour archetypes show the actual colour.

```typescript
interface Archetype {
  id: string;                  // e.g. "medium-wood", "bold", "soft-texture"
  category: SurfaceCategory;
  label: string;               // human-readable, shown in UI
  image: string;               // representative photo
  value: "light" | "medium" | "dark";
  textureType: TextureType;    // the physical substance — drives harmony
  texture: TextureLevel;
}
```

**Collections are swappable when they share the same archetype ID in their pool.** This is the only rule needed for collection matching.

### Layer 3: Collection
**Job:** Guarantee real-world harmony. Every material combination within a collection has been physically verified by a designer.

Collections are **not generated by tags**. They are manually curated. Tag-based matching alone is not sufficient for designer-level harmony — two materials can both be tagged "warm" and still clash in reality.

**A collection is a curated set of real material IDs** — not archetype IDs. Each real material in the collection is tagged with its `archetypeId` and `category`. This is how the system knows which collection matches the user's archetype choices: it finds the first collection whose materials cover all the chosen archetypes across their respective categories.

```
User picks: vibe=warm-grounded + floor=medium-wood + cabinets=light-wood
→ system scans collections in array order
→ finds first collection that contains:
    a material with archetypeId="medium-wood" in category="flooring"
    AND a material with archetypeId="light-wood" in category="cabinet-fronts"
→ that collection wins
→ its material photos are shown in the moodboard tiles
```

**Collection priority** is determined by the order of the collections array — the first matching collection wins. To promote a collection, move it earlier in the array. No numeric rank field needed.

```typescript
interface Collection {
  id: string;
  name: string;
  vibe: VibeTag;
  thumbnail: string;
  promptBase: string;              // for AI visualization
  materials: {
    flooring: string[];            // real material IDs — designer-verified
    "cabinet-fronts": string[];    // any 1 or 2 from this list work together
    "worktops-and-backsplashes": string[];
    "tiles": string[];
    "walls": string[];             // collection default, not user-chosen
    "accents": string[];           // collection default, not user-chosen
  };
}
```

**How collection matching works:**
The system checks whether a collection contains at least one material per chosen archetype+category pair. All materials in the collection have been physically verified to work together — so any combination the user builds within a collection is guaranteed harmonious.

**Cabinet fronts:** The user picks any 1 or 2 materials from the collection's cabinet-fronts list. No primary/secondary distinction — any combination within the list is guaranteed to work together.

**Walls and accents** are part of the collection's material list and shown on the moodboard, but they are not used as matching signals. They complete the visual composition automatically.

**Two types of swapping once a collection is matched:**

1. **Swap material within collection** — user picks a different material from the same slot's list within the current collection. Same harmony guarantee, different product. This is the main way users personalise their moodboard.

2. **Swap collection** — user changes an archetype choice, triggering a new collection match. Compatible slots carry over their material where possible.

**Collections grow over time.** Each new collection automatically enriches the system. No code changes required — only data.

### Layer 4: Real Materials
**Job:** Handle price tiers, material type preferences (vinyl vs engineered wood), and supplier data. Invisible to the harmony system.

```typescript
interface RealMaterial {
  id: string;                      // e.g. "mat-flooring-047"
  name: string;                    // e.g. "Boen Oak Animoso"
  category: SurfaceCategory;
  archetypeId: string;             // links up to archetype
  value: "light" | "medium" | "dark";
  tone: "cool" | "warm" | "natural" | "neutral";
  texture: TextureLevel;
  textureType: TextureType;        // wood / stone / marble etc. — harmony system
  materialType: MaterialType;      // engineered-wood / vinyl / porcelain etc. — shopping only
  priceTier: "budget" | "mid" | "premium";
  price: number;
  supplier: string;
  inStock: boolean;
}
```

**Real material images appear in Phase 1** — once the collection matches after the 2nd archetype pick, tiles update to show the real material photo from the winning collection. What is exclusive to Phase 2 is real material *alternatives* — swapping between products that share the same look but differ in price tier, brand, or material base.

## Type Definitions

```typescript
type SurfaceCategory =
  | "flooring"
  | "cabinet-fronts"
  | "worktops-and-backsplashes"
  | "tiles"
  | "walls"
  | "accents";

// The physical substance — drives harmony and visual matching
type TextureType =
  | "wood"
  | "stone"
  | "marble"
  | "concrete"
  | "painted"
  | "metal"
  | "ceramic";

// The product base — drives shopping filters only, invisible to harmony system
type MaterialType =
  | "engineered-wood"
  | "solid-wood"
  | "vinyl"
  | "porcelain"
  | "natural-stone"
  | "quartz"
  | "mdf-painted"
  | "lacquered";

type TextureLevel =
  | "smooth"      // no visible grain or pattern
  | "subtle"      // very light grain or texture
  | "medium"      // visible grain or pattern
  | "rich"        // strong grain, texture, or veining
  | "statement";  // dominant visual — heavy veining, bold grain
```

## User Journey

The journey has two distinct phases with different emotional registers: **designing** and **shopping.**

### Phase 1 — Visual Composition

```
STEP 1: VIBE PICK
  User sees 3 full-screen mood images
  Picks one: Light & Airy / Warm & Grounded / Bold & Moody
  → narrows the active collection pool
  → no materials shown yet

STEP 2: FIRST MATERIAL PICK
  User taps a surface slot on the moodboard
  Bottom sheet shows archetypes for that category filtered by vibe
  User picks one → tile appears showing the archetype image
  → collection matching does not trigger yet — not enough signal

STEP 3: SECOND MATERIAL PICK
  User picks a second archetype from any slot
  → system now has: vibe + 2 archetypes
  → collection matching triggers for the first time
  → system finds the first collection in the array containing both archetypes
  → both tiles update from archetype images to real material photos
     from the winning collection
  → moodboard begins to feel real and specific

STEP 4: SUBSEQUENT PICKS
  Each additional archetype choice either confirms the current collection
  or shifts it to the next best match in the array
  → tiles already showing real material photos may update if collection shifts
     this should feel like "focusing", not breaking
  → walls and accents appear automatically as collection defaults
  → user can step back at any time — collection re-evaluates on each change

STEP 5: MOODBOARD COMPLETE
  All slots filled
  All tiles show real material photos from one confirmed collection
  Walls and accents visible as collection defaults
  User reviews the full visual composition and confirms that it is finished
  → "Lock moodboard" transitions to Phase 2
```

### Phase 2 — Product Refinement

```
STEP 6: SLOT-BY-SLOT PRODUCT ALTERNATIVES
  User goes through each slot to select the actual product
  For each slot they see real material alternatives:
  — all verified options within this slot from the confirmed collection
  — filtered by material type (vinyl / engineered wood / solid / etc.)
  — filtered by price tier (budget / mid / premium)
  Swapping here never breaks harmony — all alternatives are pre-verified

  If the user wants a fundamentally different direction they return to
  Phase 1 and adjust their archetype choices. Phase 2 is purely product
  refinement within the locked visual composition.
```

## What the System Must Never Do

- **Never use tag matching alone to guarantee harmony.** Tags help curators find candidates. Only manual curation and collection membership confirms a match.
- **Never show the user words like "warm undertone", "cool tone", "neutral".** These are internal system concepts only.
- **Never show more than ~6 options in a single archetype picker.** If a category has more archetypes, filter by vibe first.
- **Never break harmony when switching collections.** Always carry over compatible slots. Never leave the user with a visually incoherent moodboard.
- **Never auto-generate collections from tags.** Collections are always manually curated.
- **Never show real material photos before the second archetype is picked.** The archetype image must be shown first.
- **Never surface collection matching or array ordering as visible UI.** The moodboard should feel like it is naturally evolving toward the user's taste.
- **Never distinguish between primary and secondary cabinet fronts.** The collection pool defines valid options; the user chooses freely within it.
- **Never offer collection switching in Phase 2.** If the user wants a different direction, they return to Phase 1.

## Curation Workflow

Collections, archetypes, and materials are maintained manually by editing TypeScript data files directly. There is no admin UI.

The system grows by adding:
1. **New real materials** — tagged with `archetypeId`, `textureType`, and `materialType`
2. **New collections** — pool of verified archetype IDs per slot, positioned in the array to reflect priority
3. **New archetypes** — only when a genuinely new material type is needed that existing archetypes cannot represent

Adding a new collection automatically enriches swap options for any user whose archetype choices match. No code changes required — only data.
